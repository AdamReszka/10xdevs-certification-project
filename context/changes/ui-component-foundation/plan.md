# UI Component Foundation (F-03) Implementation Plan

## Overview

Deliver the remaining two-thirds of F-03 (`ui-component-foundation`). The shadcn/ui *infrastructure* is already wired and verified (the resolved Unknown); the actual UI has not been built. This plan: (1) adds the shadcn component set and reconciles the `baseColor` config/doc discrepancy, (2) builds a full styled app shell + base layout, and (3) creates the three auth page shells at `/login`, `/signup`, `/reset` with placeholder content ready for S-01 to populate. Completing this slice proves the shadcn + Tailwind CSS 4 wiring renders with no style regression — the roadmap's explicit gate.

## Current State Analysis

From `context/changes/ui-component-foundation/research.md`:

- **shadcn/ui infra is fully wired** — `components.json` (new-york, `baseColor: zinc`, lucide), `shadcn@^4.8.3`, `src/app/globals.css:1-2` imports `tailwindcss` + `shadcn/tailwind.css`, full OKLCH token set (`:root` + `.dark` + `@theme inline` + `@layer base` reset), `src/lib/utils.ts` `cn()`. Supporting deps (`class-variance-authority`, `clsx`, `tailwind-merge`, `tailwindcss-animate`, `lucide-react`) installed.
- **`src/components/ui/` is empty** — no shadcn component has ever been added; the roadmap's "render one component, verify no regression" gate is unmet.
- **`src/app/layout.tsx` is the unmodified create-next-app default** — `metadata` is `"Create Next App"` / `"Generated by create next app"`; Geist fonts wired (theme references `--font-geist-sans`/`--font-geist-mono`); no nav/shell.
- **`src/app/page.tsx`** is a hand-built marketing landing using raw Tailwind + raw `<a>` tags (no shadcn).
- **Atomic-design folder tree** scaffolded (`atoms/`, `molecules/`, `organisms/{anomaly,auth,dashboard,setup}/`, `providers/`, `templates/`) — all empty (`.gitkeep`).
- **Auth backend exists (F-01)** — `middleware.ts`, `src/lib/auth.ts` (`requireSession()`), `src/app/api/auth/[...all]/route.ts`, `better-auth@^1.6.12`. No Better Auth *client* file exists yet.
- **Route names are contracted** — `middleware.ts:21` whitelists exactly `/login`, `/signup`, `/reset` (+ `/api/auth`) as public prefixes. The pages don't exist → they 404 today.
- **Middleware gates `/`** — the matcher covers `/`, and `/` is not in `PUBLIC_PREFIXES`, so an unauthenticated visit to the landing page redirects to `/login` (which 404s today). The public marketing landing is currently unreachable.

## Desired End State

- `npx shadcn add` has populated `src/components/ui/` with the component set; `npm run build` and `npm run lint` pass.
- Visiting `/` shows the SprintFlow landing page rendered inside a **full styled `AppShell`** (branded nav bar with nav links + *Sign in* / *Get started* actions, `<main>`, footer), with shadcn components (Button, etc.) styled correctly in light, and in dark when `class="dark"` is applied to `<html>` — no style regression. (Automatic OS-driven theme switching is out of scope — see below.)
- Visiting `/login`, `/signup`, `/reset` shows **centered placeholder auth-card forms** (Card + Input + Label + Button) — no redirect, no 404 — visibly ready for S-01 to wire to Better Auth.
- Root-layout metadata reads SprintFlow (not "Create Next App"); a Sonner `<Toaster />` is mounted app-wide.
- `components.json` keeps `baseColor: zinc`; `CLAUDE.md` is corrected to say "zinc base" (matching reality).

Verify by: `npm run build` + `npm run lint` green, then manually loading `/`, `/login`, `/signup`, `/reset` in `npm run dev`.

### Key Discoveries:

- shadcn add works against the native TW4 path (`shadcn/tailwind.css`); no `tailwind.config` (`components.json` `tailwind.config: ""`) — tokens live in `globals.css` under `@theme inline`.
- The `(auth)` route group yields URLs `/login`, `/signup`, `/reset` (route groups don't affect the URL) — matches the `middleware.ts` public-prefix contract exactly.
- `middleware.ts` `isPublic()` uses `pathname === p || pathname.startsWith(p + "/")`; adding `"/"` to `PUBLIC_PREFIXES` matches **only** exactly `/` (since `startsWith("//")` won't match real paths) — a safe way to make the landing page public without opening child routes.

## What We're NOT Doing

- **No auth wiring** — forms are placeholder shells with no submit handlers, no Better Auth client calls, no validation. S-01 owns sign-up/sign-in/reset behavior.
- **No authenticated nav variant** — the `AppShell` nav shows *Sign in* / *Get started* (public). The sign-out / user-menu variant (needs a session) is deferred to S-01/S-07.
- **No dashboard or setup pages** — those are S-04/S-07/S-10.
- **No test framework** — per decision, verification is build + lint + manual visual; installing Vitest/Playwright is out of scope for this foundation slice.
- **No data model, API routes, or business logic.**
- **No `baseColor` token regeneration** — keeping the installed `zinc` tokens; only the CLAUDE.md doc line changes.

## Implementation Approach

Three sequential phases, each independently buildable. Phase 1 lands the component primitives (and proves they compile). Phase 2 builds the app shell and makes the landing page render inside it (proving the components render *styled* — the regression gate). Phase 3 adds the auth route group and three placeholder forms. Components follow the mandated atomic-design taxonomy (CLAUDE.md): shadcn primitives in `ui/`, custom stateless bits in `atoms/`, composite nav in `molecules/`, feature sections in `organisms/`, page-level shells in `templates/`.

## Critical Implementation Details

- **Route group URLs**: the auth pages MUST resolve to exactly `/login`, `/signup`, `/reset` to match `middleware.ts:21`. Use a `src/app/(auth)/` route group (parentheses = no URL segment) so the shared centered layout doesn't alter the paths.
- **Landing page reachability**: `/` is currently gated. Add `"/"` to `PUBLIC_PREFIXES` in `middleware.ts` — relying on the exact-match behavior of `isPublic()` so no child routes are inadvertently opened.
- **Keep Geist fonts**: `globals.css` `@theme inline` references `--font-geist-sans`/`--font-geist-mono`; do not remove the font variables from the root layout or the theme breaks.

## Phase 2: App shell + base layout

### Overview

Build the full styled `AppShell` (nav + main + footer), update root-layout metadata, mount the Sonner toaster, make `/` publicly reachable, and retrofit the landing page to render inside the shell. Loading `/` after this phase proves shadcn components render *styled* with no regression.

### Changes Required:

#### 1. Brand atom

**File**: `src/components/atoms/brand.tsx`

**Intent**: Stateless SprintFlow wordmark/logo lockup reused by the nav (and later auth shell). A lucide icon + "SprintFlow" text wrapped in a link to `/`.

**Contract**: Default-exported presentational component, no props required (optional `className`). Uses `cn()` and a lucide icon.

#### 2. Main nav molecule

**File**: `src/components/molecules/main-nav.tsx`

**Intent**: The horizontal nav-link group for the app shell header (e.g., placeholder links: Dashboard, Refinement — non-functional `href="#"` placeholders, since those routes land in later slices).

**Contract**: Presentational; renders a list of `{ label, href }` items. Marked client-only only if it needs `usePathname` for active styling — prefer a server component with static links for F-03.

#### 3. AppShell template

**File**: `src/components/templates/app-shell.tsx`

**Intent**: Page-level shell: a sticky branded header (Brand + MainNav + right-side action slot) over a `<main>` content region with a simple footer. The right-side actions are passed in (children/prop) so the landing page supplies *Sign in* / *Get started* buttons (`Button` with `asChild` wrapping `next/link`) — the authenticated sign-out/user-menu variant is added in S-01.

**Contract**: `AppShell({ children, actions }: { children: ReactNode; actions?: ReactNode })`. Header uses `Separator`/border tokens; layout fills height (root `body` is already `min-h-full flex flex-col`). Built from `ui/button`, `Brand`, `MainNav`.

#### 4. Root layout metadata + Toaster

**File**: `src/app/layout.tsx`

**Intent**: Replace the create-next-app metadata with SprintFlow's, and mount the Sonner `<Toaster />` app-wide so future toasts work. Keep the Geist font variables intact.

**Contract**: `metadata.title` → "SprintFlow", `metadata.description` → product one-liner. Add `<Toaster />` (from `@/components/ui/sonner`) inside `<body>`. Do not remove `geistSans`/`geistMono` variable wiring.

#### 5. Make landing public + retrofit into AppShell

**File**: `middleware.ts`, `src/app/page.tsx`

**Intent**: Add `"/"` to `PUBLIC_PREFIXES` so the marketing landing is reachable unauthenticated. Wrap the existing landing content in `<AppShell>` and replace the two raw `<a>` CTA buttons with shadcn `Button` (`asChild` + `next/link` / external `<a>`), and supply *Sign in* / *Get started* as the shell's `actions`.

**Contract**: `middleware.ts` `PUBLIC_PREFIXES` gains `"/"` (exact-match safe per Critical Details). `page.tsx` renders `<AppShell actions={…}>…</AppShell>`; existing copy preserved; CTAs become `Button` variants (e.g., default + outline) — at least one real shadcn `Button` rendered on the page (the gate).

## Phase 3: Auth page shells

### Overview

Create the `(auth)` route group with a shared centered auth-card layout and three placeholder forms at `/login`, `/signup`, `/reset`. Forms are visual shells (no submit logic) ready for S-01.

### Changes Required:

#### 1. Auth layout (route group)

**File**: `src/app/(auth)/layout.tsx`

**Intent**: A minimal centered layout for auth pages — vertically/horizontally centered single column, no app nav (auth pages shouldn't show the dashboard nav). Brand lockup at top.

**Contract**: Server component; centers children in a full-height flex container; renders `Brand` above the card slot. Route group `(auth)` keeps child URLs at `/login` etc.

#### 2. Login / Signup / Reset form organisms

**File**: `src/components/organisms/auth/login-form.tsx`, `signup-form.tsx`, `reset-form.tsx`

**Intent**: Placeholder form shells composed from `Card` (header/title/description/content/footer), `Label` + `Input` fields, and a `Button` — visually complete but inert (no `onSubmit`, no validation). Login: email + password + submit + links to signup/reset. Signup: email + password + confirm + submit + link to login. Reset: email + submit + link to login.

**Contract**: Each is a presentational component (can stay server components — no interactivity yet). Inter-page links use `next/link`. No Better Auth client import. A `// TODO(S-01): wire to Better Auth` comment marks the submit boundary.

#### 3. Auth route pages

**File**: `src/app/(auth)/login/page.tsx`, `src/app/(auth)/signup/page.tsx`, `src/app/(auth)/reset/page.tsx`

**Intent**: Thin route files that render the corresponding form organism inside the `(auth)` layout, each with a per-page `metadata.title`.

**Contract**: Each default-exports a component rendering its form organism; sets `export const metadata` title (e.g., "Sign in · SprintFlow").
