## Research Question

Next on the roadmap is F-03 (`ui-component-foundation`). Several UI things are already installed. Determine what is already done and what is still missing for F-03, comparing the current codebase against the roadmap's stated outcome.

## F-03 target (from `context/foundation/roadmap.md:107-119`)

> **Outcome:** (foundation) shadcn/ui installed and configured for Tailwind CSS 4; base layout component (nav, main, page shell); auth page shells (`/signup`, `/login`, `/reset`) with placeholder content ready for S-01 to populate.

Three deliverables: **(1)** shadcn/ui infra, **(2)** base layout component, **(3)** three auth page shells. Roadmap risk note: *"Integration wired but no component rendered yet — add one shadcn/ui component (e.g. Button) to a page and verify no style regression before marking F-03 done."*

## Summary

F-03 is roughly **one-third done**: the shadcn/ui *infrastructure* is fully wired and verified (the resolved Unknown), but **no actual UI has been built yet**. Of the three deliverables, only #1 (infra) is in place. #2 (base layout component) and #3 (auth page shells) are entirely missing, and the roadmap's "render one component to prove no style regression" gate has not been exercised — `src/components/ui/` is empty.

Importantly, the **auth backend already exists** (landed in F-01): `middleware.ts`, `src/lib/auth.ts`, and `src/app/api/auth/[...all]/route.ts` are present, and the middleware already whitelists `/login`, `/signup`, `/reset` as public prefixes — but the pages those routes point to **do not exist yet**, so they currently 404. F-03's job is to create those shells.

## Detailed Findings

### Deliverable 1 — shadcn/ui infrastructure: DONE ✅

This is the Unknown the roadmap already marked resolved (`roadmap.md:116-117`), and the codebase confirms it.

- **`components.json`** present — `style: new-york`, `rsc: true`, `tsx: true`, `baseColor: zinc`, `cssVariables: true`, aliases (`@/components`, `@/lib/utils`, `@/components/ui`, `@/components`, `@/hooks`), `iconLibrary: lucide`.
- **`src/app/globals.css:1-2`** imports `tailwindcss` then `shadcn/tailwind.css` — the native TW4 integration path (no shim needed).
- **Full OKLCH token set** present: `@theme inline` mapping (`globals.css:6-46`), `:root` light tokens (`:48+`), `.dark` block (`globals.css:82`), and the `@layer base` reset applying `border-border` / `bg-background text-foreground` (`globals.css:117-124`). Includes card/popover/primary/secondary/muted/accent/destructive/border/input/ring, chart-1..5, and sidebar tokens.
- **`src/lib/utils.ts`** exports `cn()` via `clsx` + `tailwind-merge`.
- **Supporting deps installed** (`package.json`): `class-variance-authority@^0.7.1`, `clsx@^2.1.1`, `tailwind-merge@^3.6.0`, `tailwindcss-animate@^1.0.7`, `lucide-react@^1.17.0`, with `shadcn@^4.8.3` + `tailwindcss@^4` + `@tailwindcss/postcss@^4` as devDeps.
- **Atomic-design folder tree** scaffolded — `src/components/{atoms,molecules,organisms/{anomaly,auth,dashboard,setup},providers,templates}/` all present (each holds a `.gitkeep`). `src/components/ui/` exists but is **empty**.

### Deliverable 2 — base layout component: MISSING ❌

- **`src/app/layout.tsx` is still the unmodified create-next-app default.** Geist fonts are wired (and the `--font-geist-sans` / `--font-geist-mono` vars are referenced by the theme in `globals.css`), but:
  - `metadata` is still `title: "Create Next App"`, `description: "Generated by create next app"` — placeholder text to replace.
  - There is no nav, no app shell, no page-shell structure — just `<html><body>{children}</body></html>`.
- **`src/components/templates/` is empty** (`.gitkeep` only) — the page-level shell organism the roadmap calls for doesn't exist.
- `src/app/page.tsx` is a hand-written marketing landing page using **raw Tailwind utility classes and raw `<a>` tags — no shadcn components** (e.g. the CTA "buttons" are styled `<a>` elements, not the shadcn `Button`). This is the file most natural to retrofit with a real shadcn component to satisfy the roadmap's "render one component" gate.

### Deliverable 3 — auth page shells: MISSING ❌

- **No route files exist** for the three auth pages. `src/app/` contains only `layout.tsx`, `page.tsx`, `globals.css`, `favicon.ico`, and `api/auth/[...all]/route.ts`. There is no `src/app/login/`, `src/app/signup/`, or `src/app/reset/` (and no `(auth)` route group).
- **`src/components/organisms/auth/` is empty** (`.gitkeep` only).
- **The routes are already expected by the gate**: `middleware.ts:21` declares `PUBLIC_PREFIXES = ["/login", "/signup", "/reset", "/api/auth"]`. So the middleware lets these paths through unauthenticated — but with no page files they 404 today. F-03 creates the placeholder shells; **S-01 populates them with the real auth forms** (per `roadmap.md:130`, S-01 depends on "F-03 auth page shells").

## Architecture Insights

- **Tailwind CSS 4 + shadcn integration is the native `shadcn/tailwind.css` path** — no `tailwind.config` file (`components.json` `tailwind.config` is `""`), tokens live in `globals.css` under `@theme inline`. Adding components via `npx shadcn add <name>` (per CLAUDE.md / memory) should "just work" against this setup; the gate is to prove it once.
- **Atomic design is the mandated component taxonomy** (CLAUDE.md): shadcn primitives → `src/components/ui/`; custom stateless → `atoms/`; composites → `molecules/`; feature sections → `organisms/{anomaly,dashboard,auth,setup}/`; page shells → `templates/`; context wrappers → `providers/`. The base layout component (deliverable 2) belongs in `templates/`; auth shells (deliverable 3) compose `organisms/auth/` into route files.
- **Route naming is already contracted by F-01's middleware** — the shells must live at exactly `/login`, `/signup`, `/reset` (not `/sign-in` etc.) to match `PUBLIC_PREFIXES`.

## Discrepancies / risks to confirm at plan time

1. **`baseColor` mismatch.** `CLAUDE.md` states shadcn was set up with a **neutral** base; `components.json` says `"baseColor": "zinc"`. The OKLCH grayscale tokens in `globals.css` are achromatic (chroma 0) so the visual difference is negligible, but the config string and the documented intent disagree — decide which is canonical before adding components (shadcn reads `baseColor` when generating new component CSS). Low impact, worth a one-line resolution.
2. **`lucide-react@^1.17.0`** is an unusual version pin (lucide-react's mainline is `0.x`). It installs and is the configured `iconLibrary`; flag to verify icons render when the first component using one is added. Non-blocking.
3. **Roadmap gate is explicit and currently unmet**: render at least one shadcn component (Button suggested) on a page and confirm `npm run build` + visual styling has no regression *before* marking F-03 done. `src/components/ui/` being empty means this hasn't happened.
4. **`layout.tsx` placeholder metadata** ("Create Next App" / "Generated by create next app") should be replaced as part of the base-layout deliverable.
5. **No test framework** (CLAUDE.md) — acceptable for a pure UI-scaffold foundation; verification here is build + visual, not unit tests.
